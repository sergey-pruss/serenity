Serenity: что это, как устроено, куда ходят URL

Один репозиторий со статическим снимком нового сайта (главная и перенесённые страницы). Прод постепенно переводится с WordPress на эту статику через Nginx — без конфликта путей с темой WP.

Зачем это нужно

Цель: быстро отдавать перенесённые страницы как статический HTML/CSS/JS с одного каталога на сервере, не ломая старый сайт на WordPress там, где страницы ещё не перенесены.

Источник правды по вёрстке: прод https://serenity.agency (и оригинал дизайна там же, где он зафиксирован в задаче).

  • Новые страницы — из репозитория → сборка → /var/www/static → три URL получают один и тот же набор файлов.
  • Legacy (WP) остаётся за прокси для всех путей, которые ещё не переключены в карте маршрутизации.

Правила работы (кратко для команды и агентов)

Этот файл объясняет маршрутизацию и инфраструктуру. Ниже — «человеческий» конспект обязательного регламента задач без юридического тона; полный авторитетный текст — в корне репозитория.

  • Перед принятием задачи нужны осмысленные автоматические проверки; одноразовые тесты под задачу удаляют после того, как результат зафиксирован в истории версий.
  • Внешний вид и контент сверять с живым продом serenity.agency; копировать пиксель в пиксель, не добавлять от себя.
  • В описаниях коммитов использовать русский язык.
  • После готовности выкладывать статику на новый сервер статического контура (IP и роль — в AGENTS.md) по регламенту AGENTS.md: deploy.sh, при смене nginx/routing.conf и vhost — скрипты оттуда же; старый WordPress-хост из репозитория не деплоим. Worker — отдельно: npx wrangler deploy.
  • Стили и скрипты с долгим кэшем живут под /_sa/: если их трогали, обновить маркер версии в запросах (?v=… в шаблоне) и заново собрать HTML так, как принято в репозитории.
  • Внутри таблиц стилей пути к картинкам и шрифтам задают относительно самого файла CSS; обход через «двойной» /_sa/ даёт ошибки загрузки.
  • Дополнительно по карточкам кейсов и CDN — см. конспект в AGENTS.md.
  • По желанию после выхода на прод можно прогнать npm run test:post-deploy-smoke, если среде доступен интернет.

Единый регламент агентов и выкладки: AGENTS.md. Контекст инфраструктуры, интеграций и регламент команд — в AGENTS.md; ниже — URL, индексирование и аналитика без дублирования полного чеклиста деплоя.

Локальная разработка и прод

Команда npm run dev запускает node scripts/dev-server.cjs. Перед стартом скрипт собирает данные кейсов и главную (build-cases-all-data, build-case-all-pages, assemble-html build), затем поднимает HTTP-сервер на 127.0.0.1: по умолчанию порт 8765 (переменная окружения PORT переопределяет его; если 8765 занят, скрипт перебирает следующие порты до 8815).

  • Совпадает со статикой: раздаются уже собранные index.html, case/all/…, файлы под /_sa/ после той же цепочки сборки, что и для артефакта деплоя.
  • Не совпадает с продом: нет Nginx-карты маршрутов и прокси на WordPress — локально вы не увидите legacy-страницы по тем же правилам, что на serenity.agency. Заголовки ответа у дев-сервера — no-store (удобно для разработки); на проде HTML и часть путей кэшируются иначе.
  • Форма заявки (POST /api/lead): на стейджинге Worker (*.workers.dev) запрос обрабатывает Worker (src/worker.mjs → lead-api.mjs). На основном домене в продовом vhost (nginx/serenity-router.live.conf) весь префикс /api/ проксируется на legacy origin — поведение формы там зависит от того, что отвечает старый контур, а не Worker. Для проверки интеграций с Resend/Amo ориентируйтесь на окружение из AGENTS.md и фактическую конфигурацию vhost.

Локальный npm run dev не заменяет проверку на реальных URL с CDN/кэшем — см. AGENTS.md.

Три поверхности и один артефакт

URL Назначение
https://serenity.agency Основной домен: Nginx отдаёт новую статику по правилам карты + прокси на legacy для остального. Путь /api/* в продовом vhost (nginx/serenity-router.live.conf) сейчас уходит на legacy origin, а не на Worker.
https://static.serenity.agency Превью того же root (удобно проверять чистую статику).
https://serenity.sergeyprus.workers.dev Cloudflare Worker: те же файлы как ASSETS; API форм и т.п. — по путям /api/* на Worker.

На сервере каталог /var/www/static общий для основного домена и static-поддомена — после deploy.sh обновляются оба.

Домены и DNS (смысл)

Три публичных origin’а отдают один и тот же набор статических файлов из репозитория после деплоя, но окружение различается (Nginx vs Worker, кэш, прокси).

  • serenity.agency — основной сайт: Nginx подмешивает новую статику из /var/www/static по карте маршрутов и проксирует остальное на legacy WordPress. IP сервера статики и порядок выкладки — в AGENTS.md (сейчас 168.222.142.141, каталог /var/www/static).
  • static.serenity.agency — тот же дисковый root, удобно смотреть «чистую» статику без смешения с поведением основного vhost.
  • serenity.sergeyprus.workers.dev — Cloudflare Worker с привязкой ASSETS к файлам репозитория; для путей /api/* Worker выполняется первым (wrangler.jsonc, run_worker_first).

Записи DNS у регистратора направляют домены на Cloudflare или на сервер согласно вашей схеме; детали зон здесь не дублируем — при сомнениях смотрите операционный гайд AGENTS.md.

Секреты и окружение

Секреты и ключи не хранятся в git. Облачное окружение — Cloudflare Workers; значения задаются через wrangler secret put … и привязки в wrangler.jsonc.

Cloudflare Workers

  • Привязка KV: AMO_TOKENS_KV — хранение актуальных OAuth-токенов AmoCRM (см. комментарии в src/lead-api.mjs).
  • Секреты (имена без значений): ключ Resend RESEND_API_KEY; для AmoCRM — AMO_SUBDOMAIN, AMO_ACCESS_TOKEN, AMO_REFRESH_TOKEN, AMO_CLIENT_ID, AMO_CLIENT_SECRET, AMO_REDIRECT_URI (должен совпадать с настройками интеграции в кабинете Amo).

Resend и AmoCRM

Заявка с формы уходит письмом через Resend и создаёт лид в AmoCRM (serenity.amocrm.ru) — см. AGENTS.md. При компрометации или плановой смене паролей интеграции — выпустить новые токены в Amo, обновить секреты в Worker и при необходимости содержимое KV; конкретные сроки ротации задаёт регламент команды, не репозиторий.

Playbook: кэш и «вижу старое»

  1. Файлы под /_sa/ на проде отдаются с длинным кэшем (immutable). Браузер или CDN может продолжать показывать прежний CSS/JS, если не сменить версию в query-string.
  2. После правок бандлов под /_sa/ поднимите маркер ?v=… в html/index.layout.html и выполните npm run build:html — см. AGENTS.md.
  3. Для кейсов и оверлея parity: синхронно обновить ссылки на стили в страницах case/all / корневом HTML при необходимости и прогнать npm run build:cases или полный build:html по пайплайну задачи.
  4. В CSS внутри url() не использовать пути вида ../_sa/… — получится /_sa/_sa/… и 404. Нужны пути относительно файла CSS, например ../img/….

Как проходит запрос (упрощённо)

Клиент GET /path
→
Nginx (serenity.agency) Смотрит URI: карта nginx/routing.conf → «новая страница» или «legacy».
→
Новое Файлы из /var/www/static (try_files).
Отдельно: всё под /_sa/ Статика нового сайта (CSS, JS, картинки, JSON) — всегда с диска static, с длинным кэшем (immutable). Не пересекается с корневыми путями WordPress (/css, /js у WP не трогаем).
Legacy Если путь не помечен как «новый», запрос уходит на старый origin (WordPress) через HTTPS-прокси — контент блога, отдельные кейсы и т.д.

Распределение URL: новое vs старое

Логика задана в репозитории в файле nginx/routing.conf (map $is_new_page). Ниже — смысл правил на человеческом языке; точные регулярные выражения смотри в файле.

Примеры путей Куда Комментарий
/ Новая статика Главная из репозитория.
/case/all, /case/all/2, /case/all/category/… (включая пагинацию и категории по шаблонам из конфига) Новая статика Каталог кейсов и фильтры — только новый контур.
/case/… (одиночный кейс), /case/all/… где второй сегмент — не число и не category по правилам карты Legacy WP Отдельные страницы кейсов пока остаются на старом сайте.
/blog, /services, прочие разделы (пока не включены в map как «1») Legacy WP Раскатка по секциям — через добавление правил в map (в конфиге есть закомментированные примеры).
/docs/… (справочник и др.) Новая статика Обычный HTML из репозитория (docs/… → после деплоя /var/www/static/docs/…), тот же артефакт, что и остальной сайт. Отдаёт Nginx на том же хосте, что и основной домен; отдельного приложения или «сервера Nuxt» под этим путём нет.
/_sa/… Новая статика Отдельный location в Nginx: не зависит от map страницы.
/api/… на основном домене Прокси на legacy (см. vhost) На Worker-стейдже API может обрабатываться иначе — см. src/worker.mjs.

Сейчас на проде: при выкладке по AGENTS.md (deploy.sh, deploy-routing.sh, deploy-serenity-router-vhost.sh) страница /docs/team-handbook.html на serenity.agency отдаётся как статический HTML с нового сервера (не Nuxt и не отдельное приложение). В robots.txt основного домена при той же конфигурации должна быть строка Disallow: /docs/ (файл robots.production.txt или fallback из корня репо).

Если снова сломалось: по /docs/… вместо файла видна ошибка legacy (WordPress, Nuxt «Server error» и т.п.) или в robots.txt нет Disallow: /docs/ — запрос ушёл не в статический root. Проверьте актуальные файлы (bash deploy.sh), карту nginx/routing.conf (~^/docs(/|$) и deploy-routing.sh), vhost nginx/serenity-router.live.conf (bash scripts/deploy-serenity-router-vhost.sh, блок location ^~ /docs/). Полный чеклист — в AGENTS.md. Ошибка на корне serenity.agency может быть связана с legacy отдельно от /docs/.

Legacy WordPress и префикс /_sa/

Публичные пути нового статического контура на проде — только с префиксом /_sa/ (на диске те же каталоги css, js, img, json внутри деплоя; Nginx мапит их под префикс). Корневые /css, /js, /img, /json не занимаем: ими пользуется тема WordPress на legacy-страницах, и перехват сломает старый сайт.

  • WordPress тема продолжает отдавать свои ресурсы с корня (/css, /js, /img…).
  • Новый сайт выносит публичные ассеты под /_sa/, чтобы не пересечься с WP и не ломать legacy-страницы.
  • В CSS внутри url() пути к картинкам/шрифтам — относительно файла CSS (например ../img/…), а не ../_sa/… (иначе браузер соберёт /_sa/_sa/… и получит 404).

Аналитика и SEO

Фрагмент с кодом счётчиков: html/partials/analytics-counters.html. Он подключается в шаблоне сборки главной (html/index.layout.html, директива partial) и копируется в страницы case/all скриптом scripts/sync-analytics-into-case-all.mjs.

Проверка «счётчик жив»: сеть в DevTools, срабатывание ym(…), при необходимости отчёт в Метрике.

Индексация и мета в HTML

В шаблоне html/index.layout.html для страниц сборки сейчас стоит <meta name="robots" content="noyaca" /> — буквально так, как закоммичено в репозитории. Это нестандартное значение для meta robots; трактовку для поисковиков здесь не задаём — при смене SEO-стратегии правят шаблон и пересобирают.

robots.txt и sitemap на проде

Ожидаемое состояние после выкладки: на https://serenity.agency/robots.txt — правила из robots.production.txt, включая Disallow: /docs/, а не короткий набор директив только от WordPress.

Если в ответе нет Disallow: /docs/ и текст как у старого WP — ответ всё ещё с legacy; повторите шаги из AGENTS.md (статика, deploy-routing.sh, vhost с location = /robots.txt).

В продовом vhost (nginx/serenity-router.live.conf) для /robots.txt задан alias на robots.production.txt (после deploy.sh оба файла лежат в статике). В nginx/routing.conf пути /robots.txt и /sitemap.xml помечены как «новая» статика (is_new_page=1), чтобы при устаревшем vhost не уходить на WordPress и не терять строку Disallow: /docs/. Корневой robots.txt в репозитории дублирует правила продакшена для этого fallback. На static.serenity.agency превью отдаёт robots.static-preview.txt (только Disallow: /), см. nginx/static.serenity.agency.live.conf.

sitemap.xml отдаётся со статического деплоя, а не из WordPress — чтобы новые маршруты вроде /case/all/… попадали в карту сайта из репозитория.

Дубликаты <title> и описаний

В отчёте Индексирование → Заголовки и описания для https://serenity.agency могут отображаться доли страниц с одинаковыми заголовками и описаниями (пример из практики: порядка 15% дублей title и 14% дублей description), группы URL с одним title (например общий «Блог — Serenity» на /blog и путях /blog/card/…).

  • Полный список URL в группе: кнопки выгрузки XLS / CSV в том же отчёте Вебмастера — материал для решений, не обязательный артефакт в репозитории.
  • Где искать причину на проде: пути /blog/card/… на основном домене отдаёт legacy WordPress (карта в nginx/routing.conf не переводит блог на статику). Уникальные title/description для этих URL настраиваются в админке WP / теме / SEO-плагине.
  • Параллельно в репозитории: шаблон листингов html/blog-index.layout.html задаёт один title «Блог — агентство Serenity» на нескольких собранных листингах — тот же класс риска, если эти файлы станут каноном для похожих URL. Отдельные статьи под blog/article/<slug>/ обычно уже с уникальным <title>.

Процесс в репозитории

  • Сообщения коммитов — на русском языке; при конфликтах веток не использовать force-push, предпочтительны revert и обычное слияние (AGENTS.md).
  • В каталоге artifacts/ могут лежить локальные снимки, отчёты Lighthouse и прочие материалы для задач; коммитить их не обязательно — по договорённости команды (в .gitignore они по умолчанию не скрыты).
